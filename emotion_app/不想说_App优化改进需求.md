# 《不想说》App 优化改进需求文档

> 目标：在现有《不想说》AI 情绪记录 App 基础上，强化“长期陪伴、情绪洞察、人生时间轴”的差异化价值，让用户感受到：这个 App 不是普通 AI 聊天工具，而是一个能帮我看见自己、理解自己、记录自己慢慢撑过来的情绪容器。

---

## 一、产品优化总方向

当前 App 已具备以下基础功能：

- 情绪倾诉
- AI 温柔回复
- 情绪总结
- 治愈卡片
- 人生时间轴
- 每日打卡
- 未来信箱
- 同频纸条

后续不要继续简单堆功能，而要围绕一个核心体验升级：

> **让用户感觉：这个 App 比我自己更懂我最近怎么了。**

普通 AI 聊天工具解决的是“我现在有人可以说话”。

《不想说》要解决的是：

> **我想知道，这一路走来，我到底经历了什么，又是怎么撑过来的。**

---

## 二、核心差异化定位

建议将产品核心定位从：

> AI 情绪记录 App

升级为：

> 一个帮你看见自己情绪规律的 AI 树洞。

或者：

> 记录那些当时说不出口的事，后来变成你撑过来的证据。

产品不应该和豆包、ChatGPT、星野等通用 AI 比“谁更会聊天”，而应该突出：

- 更私密
- 更低压力
- 更长期
- 更会整理用户自己的情绪轨迹
- 更能让用户回头看到自己走过来的证据

---

## 三、优先级最高的 3 个优化方向

### P0-1：情绪模式洞察

#### 功能名称建议

- 情绪线索
- 最近的我怎么了
- 我的情绪模式
- 情绪侦探

#### 功能目标

用户记录一段时间后，App 不只是展示日记列表，而是主动帮用户总结最近的情绪规律。

让用户看到：

- 我最近为什么总是不开心
- 哪些事情反复影响我
- 我通常在什么时候情绪变差
- 我真正需要的是什么

#### 示例输出

```text
最近 14 天，你的低落多出现在晚上 23:00 之后。

高频关键词是：工作、否定、比较、睡不着。

你最常出现的情绪不是“难过”，而是“委屈 + 疲惫”。

有 4 次记录都和“被忽视”有关。

也许你真正需要的，不是继续硬撑，而是被认真看见一次。
```

#### 页面位置建议

可以放在：

- 首页卡片
- 时间轴顶部
- 周报页面
- 个人页中的“我的情绪说明书”

#### 数据来源

基于用户已有记录分析：

- 情绪类型
- 情绪强度
- 记录时间
- 文本关键词
- AI 总结标签
- 连续打卡情况

#### Codex 实现建议

需要新增或扩展字段：

```ts
interface EmotionInsight {
  id: string;
  userId: string;
  period: '7d' | '14d' | '30d';
  dominantEmotions: string[];
  frequentKeywords: string[];
  triggerPatterns: string[];
  timePatterns: string[];
  hiddenNeeds: string[];
  summary: string;
  createdAt: string;
}
```

前端展示建议：

- 使用卡片式布局
- 不要一次展示太多文字
- 重点句子加粗或高亮
- 可以用“最近的你，好像……”作为开头

### P0-2：未来的我回信

#### 当前问题

现有“未来信箱”是一个好功能，但如果只是定时打开过去写的信，情绪冲击力还不够。

#### 优化方向

把“未来信箱”升级为：

> **未来的我写给过去的我。**

当用户打开过去写给未来自己的信时，App 不只是展示原文，而是引导用户完成一次回望。

#### 用户流程

1. 用户在低落、迷茫、焦虑时写下一封未来信。
2. 到达指定日期后，App 提醒用户打开。
3. 打开时先展示过去的原文。
4. AI 生成一段温柔的回望文案。
5. 用户补充一句“现在的我想对那时的自己说”。
6. 生成一张“我撑到了今天”的纪念卡片。

#### 示例文案

```text
这是 3 个月前的你。

那时候你很害怕，也很不确定。

但你还是撑到了今天。

现在回头看，你想对那时的自己说什么？
```

用户输入：

```text
其实你没有那么差，你只是太累了。
```

最终生成卡片：

```text
3 个月前的我，以为自己过不去了。

但今天的我想告诉那时的自己：

其实你没有那么差，你只是太累了。
```

#### Codex 实现建议

新增字段：

```ts
interface FutureLetter {
  id: string;
  userId: string;
  content: string;
  openAt: string;
  openedAt?: string;
  aiReflection?: string;
  userReplyToPastSelf?: string;
  cardImageUrl?: string;
  createdAt: string;
}
```

#### 验收标准

- 用户可以创建未来信。
- 到期后可以打开。
- 打开时展示过去内容和 AI 回望文案。
- 用户可以写一句给过去自己的话。
- 可以生成纪念卡片。

### P0-3：我撑过去的事

#### 功能目标

让人生时间轴不只是“历史记录”，而是变成用户的“生命证据”。

系统自动识别用户曾经反复提到的痛苦主题，如果后续频率下降或情绪强度变轻，就生成“撑过去了”的高光记录。

#### 示例输出

```text
你曾经连续 9 次提到“失眠”。

但最近 14 天，你只提到过 1 次。

这不是偶然，这是你慢慢走出来的证据。
```

或者：

```text
你曾经很害怕那次面试。

现在已经过去 42 天了。

当时觉得过不去的事，后来真的过去了。
```

#### 页面位置建议

- 时间轴中穿插展示
- 周报结尾展示
- 首页偶尔弹出
- 个人页单独做一个“我撑过去的事”入口

#### Codex 实现建议

新增数据结构：

```ts
interface SurvivedMoment {
  id: string;
  userId: string;
  topic: string;
  firstMentionedAt: string;
  lastMentionedAt: string;
  mentionCount: number;
  recentMentionCount: number;
  insightText: string;
  createdAt: string;
}
```

识别逻辑可以先做简单版本：

- 统计过去 30 天关键词
- 找出高频负面关键词
- 对比最近 7 天出现次数
- 如果明显减少，则生成“撑过去了”卡片

MVP 阶段不需要过度复杂，先做到“有感觉”。

## 四、其他高价值优化功能

## 1. 情绪复盘卡

### 功能目标

用户倾诉后，不只是得到一句安慰，而是得到一张结构化复盘卡。

### 卡片结构

```text
我表面上的情绪：焦虑 / 生气 / 难过

我真正难受的点：努力没有被看见

我真正需要的是：被认可、被理解、休息

今晚可以先做的一件小事：写下今天完成过的一件事

给自己的话：你不需要今晚就证明自己。
```

### 价值

很多 AI 情绪产品容易只会“抱抱你”。短期有效，长期容易腻。

情绪复盘卡的价值是：

- 帮用户看清情绪
- 帮用户找到真实需求
- 给用户一个可执行的小动作
- 让 AI 回复更像产品能力，而不是普通聊天

### Codex 实现建议

在每次倾诉完成后生成：

```ts
interface EmotionReviewCard {
  id: string;
  recordId: string;
  surfaceEmotion: string;
  realPainPoint: string;
  hiddenNeed: string;
  smallAction: string;
  selfComfortSentence: string;
  createdAt: string;
}
```

---

## 2. 情绪天气预报

### 功能目标

基于用户历史情绪规律，在容易低落的时间提前提醒用户。

这不是普通打卡提醒，而是基于用户自己的情绪数据做“预防式陪伴”。

### 示例提醒

```text
最近你连续 3 个周日晚上都容易焦虑。

今晚可能又会有一点“周一前焦虑”。

要不要提前写 3 句话，把它放下来？
```

或者：

```text
你最近 5 次低落都发生在深夜。

如果现在只是累了，不一定要相信脑子里的所有负面想法。
```

### 价值

用户会觉得：

> 它不是催我打开 App，而是在我快撑不住之前看见了我。

### Codex 实现建议

先做简单规则：

- 用户某个时间段连续多次出现负面情绪
- 系统在相近时间提前推送提醒
- 提醒文案基于历史模式生成

数据结构：

```ts
interface EmotionForecast {
  id: string;
  userId: string;
  forecastType: 'night_low' | 'monday_anxiety' | 'work_stress' | 'custom';
  triggerRule: string;
  message: string;
  scheduledAt: string;
  createdAt: string;
}
```

## 3. 我不想被安慰模式

### 功能目标

让用户在倾诉前选择自己想要的回应方式。

很多时候用户不是想听“你已经很好了”，而是想要不同类型的反馈。

### 入口设计

在输入框附近增加一个轻量选择：

```text
这次你希望我怎么回应？
```

选项：

- 只想被听见
- 想要一点安慰
- 想冷静分析
- 想要具体建议
- 想骂醒我一点
- 不要讲道理

### 回复风格示例

#### 只想被听见

```text
我在。你不用解释得很完整，也不用马上好起来。
这件事对你来说是真的很重。
```

#### 想冷静分析

```text
我们先把它拆开看：
真正让你难受的，可能不是这句话本身，而是它让你感觉自己没有被认可。
```

#### 不要讲道理

```text
好，那我不讲道理。
你今天就是很累，很委屈，也不想再努力解释了。
先把这口气放下来。
```

### Codex 实现建议

新增字段：

```ts
type ReplyMode =
  | 'listen_only'
  | 'comfort'
  | 'analysis'
  | 'advice'
  | 'wake_up'
  | 'no_reasoning';

interface EmotionRecord {
  id: string;
  userId: string;
  mood: string;
  content: string;
  replyMode: ReplyMode;
  aiReply: string;
  createdAt: string;
}
```

## 4. 我的情绪说明书

### 功能目标

记录一段时间后，为用户生成长期情绪画像。

不要做成玄学测试，而要基于真实记录。

### 示例输出

```text
你不是一个容易崩溃的人。

你更像是一个习惯忍很久，直到晚上才承认自己累的人。

你经常用“没事”盖住真正的委屈。

你最需要练习的不是坚强，而是及时求助和休息。
```

### 页面内容建议

- 我最容易低落的时间
- 我最常隐藏的情绪
- 我最需要的安慰方式
- 我最容易被什么触发
- 哪些事情真的能让我变轻松
- 最近我正在变好的地方

### Codex 实现建议

```ts
interface EmotionProfile {

  commonTriggers: string[];
  helpfulActions: string[];
  growthSigns: string[];
  profileText: string;
  updatedAt: string;
}
```

## 五、同频纸条优化建议

当前“同频纸条”方向是对的：

- 匿名
- 不能私聊
- 只能预设回应
- 不做复杂社交

后续建议继续保持克制，不要做成广场、社区、陌生人聊天。

### 推荐新增机制：今晚也有人和你一样

#### 示例文案

```text
今晚，有 17 个人也写下了“很累”。

你不是唯一一个这样的人。
```

或者：

```text
有人接住了你的纸条。

他没有评论，只给你留了一句：愿你今晚轻一点。
```

### 产品调性

重点不是社交，而是：

> 看见一点点人间回声，但不被人群打扰。

### Codex 实现建议

预设回应按钮：

- 我也有过
- 抱抱你
- 愿你今晚轻一点
- 你不是一个人
- 希望明天会好一点

数据结构：

```ts
interface ResonanceNote {
  id: string;
  userId?: string;
  content: string;
  emotionTag: string;
  isAnonymous: boolean;
  createdAt: string;
}

interface ResonanceReaction {
  id: string;
  noteId: string;
  reactionType: 'same' | 'hug' | 'lighter_tonight' | 'not_alone' | 'better_tomorrow';
  createdAt: string;
}
```

安全限制：

- 不允许私聊
- 不允许评论区
- 不展示用户资料
- 敏感内容需要审核或隐藏
- 自伤、自杀、伤害他人内容需要触发安全提示

## 六、首页体验优化

### 当前问题

如果首页只是功能列表，会让用户感觉这是一个工具集合。

但用户打开这个 App 时，大概率处在情绪混乱、疲惫、低落状态，不适合让用户做太多选择。

### 推荐首页主入口

首页主文案：

```text
今天，你想把什么先放下来？
```

下面放一个输入框：

```text
不用整理语言，想到什么写什么。
```

轻量按钮：

- 我有点累
- 我不想说
- 我想记录一下
- 我想看看最近的自己
- 写给未来的我

### 首页卡片建议

首页不要同时展示太多入口，建议只展示 3 类卡片：

1. 今日倾诉入口
2. 最近的情绪线索
3. 一条“撑过去了”的回顾

示例：

```text
最近的你，好像总是在深夜变得很累。

要不要看看这 14 天，你的情绪都在提醒你什么？
```

## 七、AI 回复风格规范

《不想说》的 AI 回复不应该像心理医生，也不应该像说教式助手。

### 回复原则

- 温柔
- 克制
- 简短
- 不评判
- 不急着给建议
- 不制造依赖
- 不假装自己是专业心理咨询师

### 避免使用

```text
你一定会好起来的
你要积极一点
你应该换个角度想
这没什么大不了
我完全理解你
作为心理咨询师
```

### 推荐使用

```text
我听见了。
这件事对你来说是真的很重。
你不用马上变好。
我们可以先把这一刻放下来。
你不是矫情，你只是撑太久了。
```

## 八、安全边界要求

《不想说》不是医疗产品，也不是心理咨询产品。

需要在产品中明确说明：

```text
本 App 不提供医学诊断、心理治疗或心理咨询服务。
如果你正在经历强烈的自伤、自杀或伤害他人的想法，请立即联系身边可信任的人，或联系当地紧急援助资源。
```

### 风险内容处理

当用户表达以下内容时，需要触发安全提示：

- 自杀意图
- 自伤计划
- 伤害他人
- 明确危险行为
- 极端绝望表达

### 处理方式

AI 回复要：

1. 先表达关心。
2. 鼓励用户立刻联系身边可信任的人。
3. 建议联系当地紧急援助资源。
4. 避免继续深度分析痛苦原因。
5. 不提供任何危险方法细节。

## 十一、最终产品体验目标

用户第一次打开时，觉得：

> 这里好像可以安全地放下一句话。

用户使用一周后，觉得：

> 它好像真的看见了我最近为什么这么累。

用户使用一个月后，觉得：

> 原来我已经撑过了那么多当时觉得过不去的事。

这是《不想说》和普通 AI 聊天产品最大的区别。

---

## 十二、给 Codex 的开发提示

请基于现有项目结构进行修改，不要大规模推翻重写。

优先从以下模块入手：

1. 首页输入与入口布局
2. 情绪记录数据结构
3. AI 回复 prompt 模板
4. 情绪复盘卡生成逻辑
5. 时间轴卡片展示
6. 未来信箱打开后的回望流程
7. 周报 / 洞察生成逻辑

开发时注意：

- 所有 AI 生成内容都要保持温柔、克制、不说教。
- 不要把产品做成心理咨询 App。
- 不要把同频纸条做成社交社区。
- 不要强迫用户打卡。
- 不要让用户感觉被评判。
- 产品重点是“记录、整理、回望、看见自己”。
