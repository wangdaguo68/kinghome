# KingModel 交易决策可视化站点设计

日期：2026-06-21

## 1. 目标

KingModel 是面向单用户的 A 股交易决策站点。系统连接已部署在阿里云服务器上的通达信 MCP，将行情、市场风格、周期、板块迁移、主线、核心标的和隔夜舆情整理为可解释的实盘决策图谱。

系统不输出黑箱买卖指令。每个结论必须同时展示数据时间、原始指标、触发规则和失效条件。

主要使用环境是电脑大屏，手机用于查看。产品同时支持盘中决策和盘后复盘。

## 2. 使用边界

- 可交易池：沪深主板和创业板。
- 默认剔除：ST、退市整理、上市不足 60 日、停牌及长期低流动性个股。
- 科创板和北交所：只作为市场风格参考，不进入候选交易池。
- V1 不使用大盘黄白线分时数据。
- V1 不依赖 120 日历史分位评分。
- 舆情只用于描述催化、共识、拥挤和预期，不直接改变仓位或市场评分。
- 不执行自动下单。

## 3. 决策链

系统按固定顺序生成结论：

> 风险硬门槛 → 赚钱/亏钱效应 → 风格识别 → 周期定位 → 板块迁移 → 主线确认 → 核心分类 → 盘中确认 → 仓位上限与失效条件

一笔候选交易必须依次通过：

> 市场允许交易 → 风格匹配 → 主线成立 → 核心出现可解释触发条件

## 4. 产品模式

### 4.1 盘中决策模式

盘中自动刷新行情和决策状态。首页优先回答：

1. 当前是否允许交易。
2. 允许使用哪种交易模式。
3. 主线和资金迁移方向是什么。
4. 三类核心候选是谁。
5. 最大负反馈在哪里。
6. 当前触发条件和失效条件是什么。

### 4.2 盘后复盘模式

收盘后生成不可变的正式快照，记录当日完整图谱、盘中状态变化、原有预判、实际结果和次日验证条件。

## 5. 信息架构

### 5.1 盘中决策台

默认首页，采用已确认的“决策驾驶舱”布局。视觉路径为：

> 交易许可 → 主线方向 → 核心标的 → 风险与失效条件

首页包含：

- 市场许可和仓位上限。
- 周期、趋势风格和投机风格。
- 赚钱效应和亏钱效应来源。
- 资金迁移图谱。
- 主线、支线及强度。
- 情绪龙头、趋势中军和弹性先锋。
- 实时风险预警。
- 舆情催化和拥挤度摘要。
- 下一时间节点的待确认事项。

### 5.2 市场图谱

展示资金从哪些板块流出、流入哪些板块，以及板块内切、横切、外切和高低切关系。节点可以进入板块详情。

### 5.3 主线与板块详情

展示板块 1 日、3 日、5 日强度，成交占比、涨停结构、梯队、分歧修复和主线评分依据。

### 5.4 核心标的详情

分别展示情绪龙头、趋势中军和弹性先锋，列出主动性、板块影响力、资金认可度、承接修复和失效条件。

### 5.5 隔夜舆情

按事件、板块和个股聚合信息，展示催化强度、跨平台共识度、拥挤度、信息增量及次日验证条件。

### 5.6 盘后复盘

保存每日正式图谱、预判与结果，支持按日期检索。

### 5.7 历史验证

回看历史判断和后续 1 日、3 日、5 日收益与回撤。历史验证用于后续人工校准规则，不是 V1 实时运行依赖。

### 5.8 系统设置

管理站点登录密码、通达信 MCP 地址、刷新频率、交易池过滤规则和舆情来源状态。

## 6. UI 设计方向

### 6.1 视觉定位

界面采用克制的工业化交易终端风格：深色背景、高对比文字、紧凑网格、有限的状态色和等宽数据字体。避免装饰性渐变、过度动效和泛用后台模板观感。

状态色含义固定：

- 红色：上涨、进攻、强势。
- 绿色：下跌、风险、负反馈。
- 黄色：等待确认、数据延迟或风险升高。
- 蓝灰色：中性信息和结构关系。

### 6.2 信息密度

- 首屏不滚动即可看到交易许可、主线、核心和风险。
- 原始指标默认折叠，通过卡片展开查看。
- 所有数据卡显示更新时间和新鲜度。
- 动效只用于状态切换和新预警出现，不持续闪烁。
- 手机端保留只读摘要、预警、主线和核心列表，复杂图谱允许横向滚动或切换为列表。

## 7. 系统架构

采用 React、FastAPI、SQLite 和 Nginx：

```text
浏览器
  ↓ HTTP 登录（测试阶段）
Nginx :19098
  ├─ React 静态站点
  └─ /api → FastAPI 127.0.0.1:19099
                   ├─ 通达信 MCP 客户端
                   ├─ 指标与状态机引擎
                   ├─ 舆情聚合器
                   ├─ 定时采集任务
                   └─ SQLite 历史快照
```

FastAPI 不直接暴露公网。浏览器只访问 19098，由 Nginx 代理 API 和 WebSocket。

SQLite 使用 WAL 模式，满足单用户读写和快照存储。后续只有在并发用户或数据规模显著增加时才迁移 PostgreSQL。

## 8. 后端模块边界

### 8.1 MCP 客户端

负责 MCP 初始化、会话续期、工具调用、分页、限速、超时、重试和响应结构校验。不得把空响应转换为 0。

### 8.2 采集调度器

按数据类型分组执行任务，避免一次拉取全市场导致 MCP 会话过期或触发频率限制。

### 8.3 数据清洗器

统一证券代码、日期、数值类型和板块名称，执行交易池过滤和异常值标记。

### 8.4 指标引擎

计算赚钱效应、亏钱效应、趋势风格、投机风格、周期状态、板块迁移、主线强度和核心评分。

### 8.5 决策规则引擎

执行风险硬门槛，生成市场许可、允许模式、仓位上限、候选核心、触发条件和失效条件。每个结论保留规则命中记录。

### 8.6 舆情聚合器

采集公开可稳定访问的数据，执行去重、实体映射、来源分级和事件聚类。舆情结果不直接进入市场评分。

### 8.7 快照仓库

保存原始 MCP 响应摘要、标准化指标、计算结论、错误记录和正式盘后快照。

## 9. 数据模型

核心实体：

- `market_snapshots`：市场广度、成交、涨跌停和强势股反馈。
- `style_snapshots`：趋势、投机及其构成指标。
- `cycle_snapshots`：当前周期、转换原因和状态持续时间。
- `sector_snapshots`：板块强度、成交占比、梯队和修复数据。
- `sector_migrations`：资金来源、去向、迁移类型和健康度。
- `core_candidates`：核心类型、评分、证据和失效条件。
- `decision_snapshots`：市场许可、仓位上限、允许与禁止模式。
- `sentiment_items`：来源、时间、链接、摘要、实体、可信级别和热度。
- `sentiment_clusters`：事件去重结果、催化、共识、拥挤和信息增量。
- `collection_runs`：采集任务状态、耗时、错误和数据时间。
- `users`：单用户登录信息和密码哈希。

## 10. 刷新策略

- 指数、核心候选和重大风险事件：约 30 秒。
- 市场广度、涨跌停和强势股溢价：约 1 分钟。
- 板块强度和资金迁移：约 3 分钟。
- 完整市场图谱：约 5 分钟。
- 手动刷新受 MCP 限速器保护。
- 收盘后生成正式快照并停止盘中高频任务。

实际频率允许在设置页调整。系统按服务器 MCP 的实测响应时间和限额自动错峰，不保证所有指标在同一秒更新。

## 11. 评分与解释

V1 使用硬门槛和可配置规则分，不使用历史分位。

- 硬门槛阻止不适合的交易模式。
- 规则分比较赚钱效应、亏钱效应、趋势、投机和板块强度。
- 每个分数保存组成指标、权重、阈值和命中结果。
- 历史验证只用于后续人工调整阈值。

页面不能只显示总分。用户必须可以展开查看“为什么得到这个结论”。

## 12. 隔夜舆情

### 12.1 决策权限

舆情仅提供预期与拥挤度提示。次日竞价、开盘 30 分钟和资金承接确认后，相关事件才可以升级为有效行情信号。

### 12.2 处理流程

1. 采集公开且稳定可访问的信息。
2. 对相同事件去重，避免转载数量虚增热度。
3. 映射到板块和核心候选。
4. 区分公告、政策、产业新闻、机构观点和社区讨论。
5. 计算催化强度、跨平台共识度、拥挤度和信息增量。
6. 生成次日竞价、开盘 30 分钟和板块承接验证条件。

每条信息保存来源、发布时间、原始链接、关联对象、摘要、催化类型、信息增量、热度和可信级别。

### 12.3 来源阶段

- 第一阶段：通达信、东方财富公开内容和交易所公告。
- 第二阶段：雪球、同花顺和公开资讯页面。
- 第三阶段：韭研社、复盘啦等需要确认稳定访问方式的来源。

来源可信度：

> 官方公告/政策 ＞ 可核验产业新闻 ＞ 专业整理 ＞ 社区观点

需要登录、验证码或存在明显反爬限制的来源显示“未接入”。系统不绕过访问控制。

## 13. 登录与安全

- V1 为单用户登录。
- 密码使用 Argon2 哈希。
- 会话 Cookie 使用 HttpOnly 和 SameSite。
- 登录接口限制失败次数。
- MCP 地址、站点密码和其他敏感配置仅存在服务器环境文件中，不提交 Git。
- FastAPI 不直接暴露公网。

测试阶段按用户选择使用 HTTP。页面持续显示未启用 HTTPS 的安全提示。正式长期使用前应增加域名和可信 HTTPS 证书。

## 14. 异常与降级

- 每张数据卡显示数据时间和新鲜度。
- 单项接口失败时保留最后一次成功值并标记“数据延迟”。
- 数据过期达到阈值后，禁止生成新的决策结论。
- MCP 连续失败时进入黄色或红色降级状态。
- 部分数据可用时只更新对应模块，不伪造完整图谱。
- 原始响应摘要、计算结果和错误分别记录。
- WebSocket 断开时前端自动重连，并退化为定时拉取。

## 15. 部署

- 公网入口：服务器 19098 端口。
- FastAPI 使用 19099，但仅绑定本机或 Docker 内部网络。
- 使用 Docker Compose 管理前端、后端和持久化目录。
- Nginx 提供静态站点并代理 `/api` 和 WebSocket。
- 容器配置健康检查和异常自动重启。
- SQLite、日志和配置使用持久化目录。
- 保留上一版本镜像和 Compose 配置用于回滚。

部署前检查服务器现有 Docker、Nginx、端口占用、磁盘空间和通达信 MCP 可用性。不得覆盖服务器上的无关服务。

## 16. 测试

### 16.1 单元测试

- 交易池过滤。
- 赚钱和亏钱效应。
- 趋势和投机规则分。
- 周期状态转换。
- 板块迁移分类。
- 主线和三类核心评分。
- 决策硬门槛与失效条件。

### 16.2 集成测试

- MCP 初始化、续期、限速和分页。
- 正常响应、空响应、超时、结构变化和部分失败。
- SQLite 快照写入、恢复和正式收盘快照。
- 舆情去重和事件聚类。

### 16.3 前端测试

- 登录和会话失效。
- 驾驶舱初始加载和实时更新。
- 指标解释展开。
- 数据延迟和系统降级提示。
- 盘中与盘后模式切换。
- 手机只读布局。

### 16.4 部署冒烟测试

- 公网入口可访问。
- 登录成功且未授权 API 被拒绝。
- MCP 状态正常。
- 实时数据和 WebSocket 更新正常。
- 收盘快照生成。
- 容器重启后历史数据仍存在。
- 回滚流程可执行。

## 17. 验收标准

1. 用户登录后，首屏可以在不滚动的情况下看到市场许可、主线、核心和风险。
2. 所有结论可以展开查看原始指标、规则和数据时间。
3. MCP 单项失败不会被显示为 0，也不会产生虚假决策。
4. 盘中状态按设定频率自动更新，收盘后生成正式快照。
5. 可以查看历史交易日图谱及后续表现。
6. 隔夜舆情可以按板块和事件查看，并明确标注来源和验证条件。
7. 服务器重启后登录、历史快照和配置正常恢复。
8. 公网 19098 可访问，19099 不直接暴露后端服务。

## 18. 非目标

- 自动下单或券商账户接入。
- 多用户权限和团队协作。
- 绕过第三方平台登录、验证码或反爬机制。
- V1 自动训练或自动优化评分模型。
- V1 引入 PostgreSQL、Redis或微服务拆分。
- 在没有行情确认时根据舆情直接产生交易信号。
