# SDLC Skill 设计文档

## 概述

`sdlc` 是一个贯穿软件开发全生命周期的 Claude CLI 通用 skill，支持从需求分析到验收上线的完整流程。支持新项目全流程启动，也支持已有项目从任意阶段切入继续。

## 目录结构

```
~/.claude/skills/sdlc/
├── SKILL.md                    # 主编排器
└── phases/
    ├── 01-requirements.md      # 需求分析
    ├── 02-architecture.md      # 系统架构
    ├── 03-prototype.md         # 原型设计
    ├── 04-ui-design.md         # UI 设计
    ├── 05-api-design.md        # API 设计
    ├── 06-backend.md           # 后端开发
    ├── 07-frontend.md          # 前端开发
    ├── 08-code-review.md       # 代码审查
    ├── 09-testing.md           # 测试
    └── 10-release.md           # 验收上线
```

## 项目类型与阶段映射

| 项目类型 | 激活阶段 |
|---------|---------|
| 全栈 Web | 全部 10 个阶段 |
| 纯后端/API | 01, 02, 05, 06, 08, 09, 10 |
| 纯前端 | 01, 02, 03, 04, 07, 08, 09, 10 |
| CLI 工具 | 01, 02, 06, 08, 09, 10 |
| 移动端 | 全部 10 个阶段 |

## 主编排器（SKILL.md）工作流程

### 启动判断

```
新项目？
  → 询问：项目名称、一句话描述、项目类型
  → 根据类型生成激活阶段列表，展示给用户确认

已有项目？
  → 扫描 docs/sdlc/ 目录，检测已完成阶段
  → 询问：从哪个阶段继续，或重新执行某个阶段
```

### 阶段执行循环

```
for each 激活的阶段:
  1. 读取 phases/XX-xxx.md（阶段专属指令）
  2. 读取 docs/sdlc/ 下已有文档作为上下文
  3. 生成本阶段草稿
  4. 展示草稿，等待用户确认/修改
  5. 用户确认后写入 docs/sdlc/XX-xxx.md（status: approved）
  6. 展示过渡提示，自动推进到下一阶段
```

### 阶段间过渡提示格式

```
✓ [阶段名] 完成，文档已保存至 docs/sdlc/XX-xxx.md
即将进入 [下一阶段名] 阶段，继续？
```

## 阶段文件结构

每个 `phases/XX-xxx.md` 包含三个部分：

### 1. 角色定义
明确 Claude 在该阶段扮演的专家角色。

### 2. 输入上下文
声明需要读取的前序文档列表。

### 3. 生成指令
具体的生成内容、格式要求和质量标准。

## 各阶段角色与职责

| 阶段 | 角色 | 核心产出 |
|------|------|---------|
| 需求分析 | 资深产品经理 | 用户故事、功能列表、验收标准、Out of Scope |
| 系统架构 | 系统架构师 | 技术选型、架构图（文字描述）、模块划分、数据库设计 |
| 原型设计 | UX 设计师 | 页面流程、线框图（文字描述）、交互说明 |
| UI 设计 | UI 设计师 | 设计规范、色彩系统、组件清单、视觉风格 |
| API 设计 | 后端架构师 | 接口列表、请求/响应格式、错误码规范 |
| 后端开发 | 后端工程师 | 代码实现方案、关键模块说明、目录结构 |
| 前端开发 | 前端工程师 | 组件结构、页面实现方案、状态管理方案 |
| 代码审查 | 代码审查专家 | 问题清单、安全隐患、改进建议 |
| 测试 | QA 工程师 | 测试用例、测试覆盖范围、已知问题 |
| 验收上线 | PM + DevOps | 验收报告、部署清单、上线检查项 |

## 产出文档格式

存储路径：`docs/sdlc/` 目录，每阶段一个文件。

文件头部元信息：

```markdown
---
phase: 需求分析
project: 项目名称
date: 2026-05-16
status: draft | approved
---
```

用户确认后 `status` 从 `draft` 改为 `approved`。

## 上下文传递机制

每个阶段执行时，自动读取 `docs/sdlc/` 下所有已存在的文档作为上下文，确保后续阶段的产出与前序决策保持一致。例如：

- 后端开发阶段自动参考架构设计和 API 设计文档
- 代码审查阶段自动参考需求文档和架构文档
- 测试阶段自动参考需求文档中的验收标准

## 非功能要求

- 每个阶段文件独立，可单独迭代优化，不影响其他阶段
- 主 SKILL.md 保持精简，只负责编排逻辑
- 所有产出文档使用标准 Markdown，兼容 Obsidian 等工具
- 阶段文件命名使用数字前缀，确保排序一致
